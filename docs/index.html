00 Technical Reference

Pivoting & Scaling Policies for Half-Precision LU

Lightweight pivoting and exact power-of-two scaling that keep LU factorization inside the narrow FP16 dynamic range — close to partial pivoting's cost, approaching complete pivoting's stability on many (not all) problems.

01 Method set

Four policies, one inequality

Every rule is analyzed through the same per-step growth law Lk+1 ≤ (1 + μk·αk)·Lk, where μk bounds the multiplier and αk bounds the pivot-row tail. Each policy is a choice of the pair (μk, αk).

RuleμₖαₖGrowth bound ρ
PP baseline112n−1
DP Delayed Pivoting1/τ (on success)1(1+1/τ)s·2n−1−s
GP Geometric Pivoting< τ√(ck−2−1)∏(1+μk√(ck−2−1))
ScaP Scattered Pivoting112n−1
DS — Dynamic Scaling
Panel-diagonal magnitude is used as a cheap O(w) proxy for trailing-block overflow risk, triggering an exact power-of-two column rescaling — exponent only, no rounding of the significand while values stay in the FP16 normal range.
02 Results (honest)

Scaling is the robustness lever; conditioning is the limit

On the SuiteSparse-derived tests at τ = 1.01, dynamic scaling raises finite refined-error coverage across the matrix × method pairs:

Without DS26 / 63
With DS55 / 63
Not a universal fix
  • pores_2, west1505 — strongly pivot-dependent (RP / CP win).
  • b2_ss — unsolved by every tested policy, with or without DS.
  • On dense families, DS makes most cases usable; HHP21 and near-singular families stay limited by growth / conditioning, not overflow.

Takeaway the paper argues for: expose adaptive pivoting and scaling policies, not a single classical default.

03 Reproduce

From clone to figures

git clone https://github.com/<org>/halfLU.git
cd halfLU
python -m pip install -r requirements.txt   # numpy scipy pandas matplotlib

# 1. Dense synthetic sweep (GPU, native FP16)
make GPU=v100                 # builds the CUDA kernels
bash experiments/run_dense.sh # τ-sweep over n ∈ {64…2048}

# 2. SuiteSparse-derived tests (MATLAB FP16 simulation)
matlab -batch "cd experiments; run_sparse"

# 3. Regenerate every figure from results/
python figures/make_figures.py

A single driver runs any rule / regime combination:

python src/lu16_unified.py --rule GP --ds on --tau 1.01 --n 1024

Dense experiments use native FP16 arithmetic on V100 — not FP64 computed then down-cast. Iterative refinement uses FP64 residuals (50-iteration cap, stop at ‖b−Ax̂‖∞ ≤ 1e-14). Timing and batched/parallel performance are deliberately out of scope.

04 Build (GPU)

Selectable architecture, non-breaking

make GPU=v100
Volta → sm_70 · V100, V100S
make GPU=t1000
Turing → sm_75 · T1000
make GPU=a100
Ampere → sm_80 · A100
make ARCH=sm_70
Explicit compute-capability override
make sim
CPU / MATLAB path, no GPU required
make figures
Rebuild all figures from results/

Unknown GPU= falls back to sm_70 with a warning, so existing builds keep working. FP16 storage needs sm_53+; tensor-core paths need sm_70+.
